Introduction

jquery.hygrid is, you guessed it, yet another jQuery grid plugin. It was inspired by other great grids plugins like flexigrid and jqGrid.

What's the diffrence ?

jQuery UI widget

Being developped in a jQuery UI widet, it makes it easy to manipulate and extend while taking full advantage of the jQuery UI framework.

The widget also provide support for events, making it useless to provide callbacks in options. For example;

// instead of $('#demo').hygrid({ refresh: function(){ // do stuff .. } }); // you can do $('#demo').bind('refresh', function(e, widget){ // do stuff .. });
Flexible plugin architecture

The grid's core is barely 150 lines of code. Pretty much every features available are plugins, making it very easy to customize or extend.

Themeroller compatible

The grid is themable with jQuery themeroller.

Transitional

This grid wasn't created to be an official jQuery UI plugin nor to replace the jQuery UI GridTable plugin currently under developement. These are to different plugin with different goals. The author(s) of the GridTable plugin wants to do things right, which means their grid isn't a priority, so it won't be available anytime soon in the official release.

I will favorize simplicity and stability over complexity and features. That's in part for this reason that I dropped the virtual scrolling functionality and IE support. However, I'm sure that the plugin architecture makes these issues addresable.

Documented

Most of the grid I've tried are well documented (save for flexigrid), but if you wish to change something in the grid you have to modify the grid's code.. which is rarely documented. Being plugin oriented, I'll try to keep the API as well documented as I can to make it easy for developpers to extend the functionalities of this grid in a non-intrusive manner.

Domain specific

At this time, the grid doesn't support IE. However it works in all other browsers I've tested so far (Firefox, Opera, Epiphany, Chrome). I don't plan to work much on IE support as I don't need it. But the grid's plugin architecture makes it really easy to change mostly all aspects of the grid, so it's possible that eventually someone will be willing to make a IE compatibility plugin. Don't hesitate to ask me questions if you need help in your quest to fix this grid for IE, but don't count on me to do it myself. It's not something I need and I've lost enought time of my life fixing IE.

Uses a single table element

Because of virtual scrolling, most grids uses multiple tables to render the grid body along with the grid header/footer. Since this grid has no support virtual scrolling, I was able to use a single table element and keep things somewhat semantically sane.

In other words, I won't trade stability and consistency for bells and whistles. My goal is to make something lightweight and usable now, not to try to perfectly mimic desktop-like grids.

Backends

Incorporation of generic backends is planned in the developpment roadmap. Every other grid systems lets up you to job of creating your own backend. I've already coded a somewhat generic flexigrid backend for Django and I plan to port it to this grid. If I can make it flexible enough to support more than one grid system, I will try to do so. I will also most likely have to make a PHP backend soon for the demos and tests.

Is it for me ?

Only you can tell. If you like it and understand its limitations and goals, well it's for you. If you really need features like virtual scrolling or flawless IE support, I'd advise you to use another grid system.
